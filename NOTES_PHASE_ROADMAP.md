## 1. The discipline for this phase (do not drift)

- **Reconcile-to-the-master-line, or BLOCK.** Each note breakdown must tie to its master concept's value. A breakdown that ties to *itself* but not to the master line is a finding that BLOCKS — the internal-tie-that-lies. This is the phase's defining risk.
- **AI structures, deterministic code owns every number.** Notes are where a flipped sign or a missing component produces a clean, self-consistent, *wrong* answer. The AI proposes note structure (which accounts are cost vs accumulated-depreciation vs which class); code does every roll-up, net, and reconciliation.
- **Build-on-synthetic-with-a-known-answer, then verify-on-real.** Build each note against a synthetic GL whose answer we know; the real-data proof (e.g. Mobily's published note) is a **named gate**, not a skipped step. "Synthetic passes" is never "verified."
- **One note family per reviewed slice.** Never batch. The first silent mis-tie would hide under the others. Each slice leaves a permanent assertion/guard.
- **Sign convention is explicit per client, never assumed.** A roll-forward nets cost against accumulated depreciation; if a real TB/GL stores depreciation positive vs negative, the netting flips. Record at confirm, flag, never silently assume. (Open §5 question, now load-bearing inside notes.)

## 2. The key insight: 3 mechanisms, not ~25 problems

There are ~25 reconciling notes but only **3 reconciliation mechanisms**. Build the pattern per mechanism once; every note of that type then becomes cheap to add.

1. **Roll-forward / movement** (GL-driven) — opening + additions − disposals ± adjustments − depreciation/amortization → closing. Hardest. Includes the **contra/ECL variant** (a gross balance plus an allowance that nets against it, contra-by-signal).
2. **Static breakdown** — a balance or a P&L line split into named components that sum back to it. Light GL or detailed TB.
3. **Calculation** — a derived figure from declared inputs (EPS).

## 4. The per-upload AI architecture (what runs when a user uploads)

**Once per archetype, human-confirmed, then frozen — NOT regenerated per upload:**
the seed structure + its formulas. (Regenerating per upload = a different chart every time, no human in the loop = the silent-failure machine.)

**Every user upload runs only these four AI proposers — all labels/codes-only, never amounts:**
1. **parse-layout** — which column is labels, which is the period, where the data region starts (the Slice-4 capability; AI proposes layout, human confirms, code extracts the values — flexible on structure, rigid on arithmetic).
2. **archetype detector** — bank vs telecom vs … (Slice B).
3. **account classifier** — TB/GL line → master concept (Slice A.1, leaves-only, block-don't-absorb).
4. **GL-region / note-structure proposer** — for the note breakdowns (this phase).

Each proposes *structure*; deterministic code owns every number; each has a human confirm or a fail-loud.

**Inputs:** TB alone → faces. TB + GL(s) → faces **plus** the notes those GLs support. The reconciliation keystone proves each GL-built note ties back to its master line, or BLOCKs.
